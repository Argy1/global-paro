## Database Schema

All tables live in the `public` schema with Row Level Security (RLS) enabled.

### Core Tables

#### `candidates`
Stores all nurse/candidate registrations.

| Column | Type | Description |
|--------|------|-------------|
| `id` | uuid (PK) | Auto-generated |
| `user_id` | uuid | Links to `auth.users` (nullable) |
| `full_name` | text | Required |
| `whatsapp_number` | text | Required |
| `email` | text | Optional |
| `journey_stage` | enum | New → Contacted → Screening → Preparing → Ready → Placed → Closed |
| `pipeline_status` | enum | new → contacted → screened → qualified → in_process → placed → closed |
| `profession` | enum | Nurse / Midwife / Other |
| `education_level` | enum | Diploma / Bachelor / Master / Other |
| `english_level` | enum | IELTS / OET / Basic / Intermediate / Advanced / Not Yet |
| `target_countries` | text[] | Array of target destination countries |
| `created_at` | timestamptz | Auto |

#### `employer_inquiries`
Hospital/clinic inquiries from the Employer page.

| Column | Type | Description |
|--------|------|-------------|
| `id` | uuid (PK) | Auto-generated |
| `institution_name` | text | Hospital/clinic name |
| `contact_name` | text | Contact person |
| `institutional_email` | text | Official email |
| `nurses_needed` | int | Number of nurses required |
| `employer_status` | enum | New / Contacted / Meeting Booked / Closed |
| `created_at` | timestamptz | Auto |

#### `webinars`
Webinar events managed by admins.

| Column | Type | Description |
|--------|------|-------------|
| `id` | uuid (PK) | Auto-generated |
| `title` | text | Webinar title |
| `slug` | text | URL slug |
| `event_date` | timestamptz | Event date/time |
| `is_published` | boolean | Visibility toggle |
| `is_featured` | boolean | Show on homepage |
| `learn_items` | text[] | Learning outcomes |
| `cost` | text | Price string (e.g., "FREE") |

#### `webinar_registrations`
Registration records linked to webinars.

#### `chat_conversations` + `conversation_messages`
AI chat session history, linked to a `user_identifier`.

#### `chat_escalation_tickets`
Human escalation tickets created when user types "talk to human".

| Column | Type | Description |
|--------|------|-------------|
| `priority` | enum | Low / Normal / High |
| `status` | enum | Open / Assigned / In Progress / Resolved |

#### `content` + `content_items`
CMS tables for News and Success Stories articles.

#### `profiles`
Extended user profile data linked to `auth.users`.

#### `user_roles`
Role assignments separate from profiles (security best practice).

| Column | Type | Description |
|--------|------|-------------|

#### `site_settings`
Key-value store for configurable site settings (managed in admin UI).

Instagram, LinkedIn, TikTok URLs (managed in admin UI).

WhatsApp community group links (managed in admin UI).

Career pathway definitions per destination country (Singapore, USA, Canada, etc.).

#### `quickstart_chapters`
10-chapter guide content managed via admin CMS.

#### `faq_items`
FAQ entries categorized by topic.

#### `candidate_documents`
Uploaded document metadata linked to candidates.

## Authentication & Roles

Authentication is handled by Supabase Auth with email/password.

### User Roles

Roles are stored in a **separate `user_roles` table** (never in `profiles`) to prevent privilege escalation.

```sql
-- Available roles
'admin' | 'moderator' | 'user' | 'editor' | 'support_agent'
```

### Role Check Function

```sql
-- SECURITY DEFINER function to safely check roles without recursive RLS
SELECT public.has_role('<user_id>', 'admin');
```

### Access Control (React)

```tsx
// Admin-only route
<ProtectedRoute>
  <AdminDashboard />
</ProtectedRoute>

// Logged-in user route
<PortalProtectedRoute>
  <Portal />
</PortalProtectedRoute>
```

### Assign Admin Role

```sql
INSERT INTO public.user_roles (user_id, role)
VALUES ('<user_id_from_auth>', 'admin')
ON CONFLICT DO NOTHING;
```

## Edge Functions (API)

Deployed as Deno serverless functions on Supabase Edge Functions.

### `POST /functions/v1/chat`

AI chat endpoint using Google Gemini 3 Flash via Lovable AI Gateway.

**Request Body:**
```json
{
  "messages": [
    { "role": "user", "content": "How do I register?" }
  ],
  "lang": "en"
}
```

**Response:** `text/event-stream` (Server-Sent Events / streaming)

**Supported languages:** `"en"` (English) | `"id"` (Bahasa Indonesia)

**Error Responses:**
| Status | Meaning |
|--------|---------|
| `429` | Rate limited — too many requests |
| `402` | AI service unavailable |
| `500` | Server error |

**Config:** `supabase/config.toml` — `verify_jwt = false` (public endpoint)

## i18n / Translations

The app supports **English** and **Bahasa Indonesia**.

```tsx
import { useTranslation } from "@/i18n/LanguageContext";

const { t, lang, setLang } = useTranslation();
// t.common.getStarted, t.nav.home, etc.
```

Translation files:
- `src/i18n/en.ts` — English (default)
- `src/i18n/id.ts` — Bahasa Indonesia

## Deployment

### Lovable Cloud (Automatic)

All deployments are handled automatically:

1. **Frontend** — Any change in Lovable editor auto-deploys the React app
2. **Edge Functions** — Files in `supabase/functions/` are deployed automatically
3. **Database migrations** — Applied via the Lovable migration tool

### Manual Deployment (via GitHub)

1. Push changes to GitHub repository
2. Lovable detects changes and rebuilds automatically

### Custom Domain

1. Go to **Project → Settings → Domains**
2. Add your domain and configure DNS records
3. SSL is provisioned automatically

### Production Checklist

- [ ] Email verification enabled (not auto-confirm)
- [ ] All tables have RLS policies
- [ ] `LOVABLE_API_KEY` secret configured for AI chat
- [ ] Admin role assigned to at least one user
- [ ] `robots.txt` and `sitemap.xml` configured
- [ ] Google Search Console verification file present
